常见问题

为什么 UE 导出的 CSV 用 Excel 打开，所有内容都挤在第一列？

这是因为 UE 导出的 CSV 并非标准的 UTF-8 编码，而 Excel 默认使用系统编码打开（Windows 下为 GBK），导致字符错位，所有数据堆在第一列。本工具不存在此问题：你只需要把 JSON 或 CSV 拖进来，工具直接生成 .xlsx 文件，用 Excel 打开时列结构清晰、排版整齐，立即可以开始批量编辑 UE DataTable 数据，无需手动分列或调整编码。

导出的 JSON 里文字变成了 NSLOCTEXT 格式，怎么编辑？

NSLOCTEXT 是虚幻引擎的本地化标记。当 DataTable 中某一列被设为 FText 类型而非 String 时，UE 在导出 JSON 时会为每个单元格生成包含 Namespace、Key 和源文本的 NSLOCTEXT 表达式，这是 UE4/UE5 支持多语言翻译的基础。本工具在转换成 Excel 时会智能解析，只显示源文本让你编辑，Key 和 Namespace 存放在右侧附加列中自动保管。新增行时，工具会自动生成符合 UE 规范的唯一 Key，改完导回 JSON，NSLOCTEXT 结构完整还原，无需手动处理。

我在 Excel 里新增了数据行，导回 UE 会不会出问题？

不会。当你把修改后的 Excel 拖回工具时，对于新增行中填写了文本但没有指定 Key 和 Package 的单元格，工具会自动扫描表内已有的 Package 值作为参照，并生成一个标准的 32 位大写 UUID 作为 Key（格式如 32813E2549E740620926B33D5ADD8E），确保全局唯一、永不重复。这种方式与 UE 编辑器内部生成新 Key 的机制完全一致，所以新增行无论导出为 JSON 还是 CSV，都可以直接导入回 Unreal Engine，无需手动调整。

这个工具对 UE 版本有要求吗？支持哪些格式？

没有任何版本限制。本工具同时支持 UE4 和 UE5，因为它处理的是 DataTable 导出的文件格式而非引擎内部数据。输入支持 UE 导出的 JSON 和 CSV 两种格式，输出可以生成标准的 .xlsx 供 Excel 编辑，编辑完成后也支持转回 JSON 或 CSV 重新导入 UE DataTable。所有转换在浏览器本地完成，无需安装任何插件或运行时环境。
